Manual de Uso
SinalTech Multi-School

Documento operacional completo para configuracao, uso diario, administracao de usuarios, permissoes, aprovacao de mudancas de horarios, backup, auditoria e monitoramento.

Versao do documento: 2026-04-28 | Sistema: Node.js + PostgreSQL + Painel Web

1) Objetivo do sistema

O SinalTech Multi-School centraliza o controle de sinais de varias escolas em um unico painel. Cada usuario acessa somente o que seu perfil permite. O sistema registra alteracoes, gera alertas operacionais e oferece fluxo de aprovacao manual ou autoaprovacao de mudancas de horarios.

Multi-escola RBAC + permissoes por funcao Aprovacao manual / autoaprovacao Auditoria Monitoramento

4) Menus e funcoes configuraveis

Menus

dashboard, config, schools, users, audit

Cada menu pode ser habilitado/desabilitado por usuario no modal de permissao.

Funcoes (granulares)

Controle fino por recurso, por exemplo: dashboard_manual_play, config_schedule_write, config_approve_changes, config_auto_approve_changes.

5) Regras de aprovacao de mudancas

Permissoes envolvidas no fluxo

  • config_schedule_write: permite iniciar mudancas.
  • config_auto_approve_changes: publica direto sem pendencia.
  • config_approve_changes: aprova/rejeita solicitacoes pendentes.

6) Guia de uso por menu

Dashboard

  • Visualiza ultimo e proximo sinal.
  • Acionamento manual de audio (quando permitido).
  • Status do banco, alertas abertos e escolas sem horario.
  • Grafico de historico operacional (latencia, falhas de toque, alertas).
  • Filtro de escola no grafico para superadmin.

Configuracoes (horarios)

Selecione a escola no topo da secao.

Clique em Novo para cadastrar um horario.

Edite/remova sinais existentes pela tabela.

Aplique template quando quiser replicar grade.

Use backup para exportar/importar/restaurar.

Aprovacao de mudancas

  • Lista de solicitacoes por escola com diff Antes x Depois.
  • Superadmin (com permissao) pode Aprovar ou Rejeitar.
  • Sem permissao de autoaprovacao, o sistema sempre gera pendencia.

Escolas

  • Cadastro, edicao e desativacao (soft delete).
  • A seta da lista seleciona escola e leva para configuracoes.

Usuarios e permissoes

  • Criacao, edicao, ativacao/desativacao e reset de senha (superadmin).
  • Permissoes por menu e funcao com heranca do perfil + override.
  • Presets de permissao (coordenacao, portaria, secretaria).
  • Simulacao de login por usuario (somente superadmin).

Auditoria

  • Filtro por escola, usuario, acao e periodo.
  • Rastreia quem mudou o que e quando.

7) Configuracao inicial (passo a passo)

  1. Defina variaveis de ambiente (.env).
  2. Rode npm install e npm start.
  3. Acesse /api/health para validar banco.
  4. Entre com superadmin inicial (seed) se for primeiro uso.
  5. Cadastre escolas.
  6. Cadastre usuarios por escola e ajuste permissoes.
  7. Defina se cada admin escola usa aprovacao manual ou autoaprovacao.
  8. Configure horarios da primeira escola e valide dashboard.

9) Checklist de operacao diaria

Inicio do dia

  • Checar /api/health.
  • Abrir dashboard e validar alertas.
  • Confirmar escola sem horario = 0.

Durante o dia

  • Monitorar falhas de toque.
  • Aprovar pendencias (se manual).
  • Conferir auditoria de mudancas criticas.

Fim do dia

  • Conferir backup diario.
  • Revisar alertas abertos.
  • Validar historico operacional.

10) Troubleshooting rapido

  • Erro 500 em API: verificar log do `server.js` e SQL.
  • database_unavailable: revisar `DATABASE_URL` / SSL / firewall.
  • simulation_read_only: usuario esta em modo simulacao.
  • permission_denied: conferir permissao efetiva do usuario.
  • EADDRINUSE: porta local ja em uso.
